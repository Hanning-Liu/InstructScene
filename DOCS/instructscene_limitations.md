# InstructScene Limitations and Improvement Directions

This note summarizes the main drawbacks identified while inspecting the
InstructScene codebase and dataset construction pipeline.

## Main Drawbacks

### 1. Missing Architectural Context

The current semantic graph focuses on furniture objects and object-object
relations. It does not explicitly model architectural elements such as:

- Doors
- Windows
- Wall openings
- Wall segments
- Circulation paths
- Walkable free space

This is a major limitation for realistic interior layout. A bedroom design is
strongly constrained by door position, window position, wall geometry, and
accessibility. Without these elements, the generated scene may satisfy
furniture-furniture relations while still being impractical as an actual room.

Possible improvements:

- Represent doors, windows, and walls as graph nodes.
- Add relations such as `near window`, `against wall`, `not blocking door`, and
  `facing window`.
- Condition generation on floor plan geometry and architectural constraints.
- Add collision, accessibility, and circulation losses or post-generation checks.

### 2. Closed Furniture and Relation Vocabulary

The system uses fixed furniture categories, fixed spatial relation categories,
and fixed object-feature codebooks. Adding new furniture types, new relation
types, or new object feature representations would likely require substantial
dataset preprocessing and model retraining.

This makes the system relatively inflexible. For example, if we want to support
new object types such as posters, sports equipment, gaming chairs, plants,
mirrors, or decorative objects, the existing trained model cannot naturally
handle them.

Possible improvements:

- Use open-vocabulary object recognition and retrieval.
- Use text-conditioned object retrieval or generation.
- Decouple layout generation from asset vocabulary.
- Add extensible graph schemas for new object and relation types.
- Use pretrained vision-language models for zero-shot or few-shot object
  grounding.

### 3. Narrow Instruction Language

The training instructions are generated using rule-based templates from
object-object relations. They mostly describe spatial relations such as:

- `bed left of nightstand`
- `wardrobe behind bed`
- `lamp above table`

This means the model is mainly trained for relation-following instructions, not
for broad natural-language design briefs. It is unlikely to reliably understand
instructions such as:

> I am an 18-year-old high school student and I like Star Wars and badminton.
> Please design a bedroom for me.

That kind of instruction requires reasoning about lifestyle, activities, user
preferences, style, function, and furniture needs. The current model was not
trained on that type of language.

Possible improvements:

- Collect real human-written design instructions.
- Add long-form lifestyle and preference descriptions.
- Use an LLM planner to convert broad user briefs into structured layout
  constraints.
- Train with multi-sentence, multi-constraint prompts.
- Add support for global goals such as `make it spacious`, `create a study
  area`, or `maximize storage`.

## Additional Limitations

### 4. Heuristic Ground-Truth Scene Graphs

The ground-truth semantic graphs are not manually annotated. They are generated
from object bounding boxes using deterministic geometric rules. The code computes
relations from object centers, bounding box corners, distances, and vertical
overlap.

This can produce geometrically valid but semantically weak relations. For
example, `lamp behind wardrobe` may be true according to coordinates, but it may
not be a meaningful design relation.

Possible improvements:

- Use human-curated relation annotations.
- Filter unimportant or accidental relations.
- Add relation importance scores.
- Distinguish functional relations from incidental geometric relations.

### 5. Circular Instruction Construction

The training instruction is generated from the same scene graph that the model
is trained to predict:

```text
geometry -> scene graph -> templated instruction
model learns: templated instruction -> scene graph
```

This is useful for controlled training, but it does not prove robust natural
language understanding. The model learns to invert a templated instruction
generation process.

Possible improvements:

- Add human-written paraphrases.
- Add noisy, ambiguous, and varied instructions.
- Train on real design conversations or design briefs.
- Evaluate on instructions that are not generated by the same templates.

### 6. Weak Global Room Design Reasoning

The model mostly reasons through local pairwise relations. Real design often
requires global intent and functional constraints:

- Leave a clear path from the door to the bed.
- Create a study corner.
- Keep the center of the room open.
- Make the room suitable for two children.
- Put the desk near natural light.

These are difficult to express using only pairwise furniture relations.

Possible improvements:

- Add room zones such as sleeping area, study area, storage area, and circulation
  area.
- Add global scene attributes.
- Add functional affordance labels.
- Combine graph generation with constraint solving or iterative refinement.

### 7. Asset Retrieval Limits Final Scene Diversity

The final 3D objects are retrieved from the available 3D-FUTURE object database.
Even if the model predicts a useful category and object feature, it cannot invent
new objects that are absent from the database.

Possible improvements:

- Integrate text-to-3D or object generation models.
- Retrieve from larger external asset libraries.
- Use generated object assets for rare or personalized requirements.
- Separate furniture layout planning from final asset synthesis.

### 8. Object Features Are Not Scene-Compatibility Features

The fVQ-VAE compresses individual OpenShape object features. These features
describe individual objects, but they do not directly encode whether a group of
objects is stylistically or functionally compatible.

Possible improvements:

- Train object features with scene-level compatibility objectives.
- Model style, color, material, and theme consistency across object sets.
- Add losses or critics for scene-level coherence.

### 9. Fixed Maximum Object Count

Bedroom generation uses a fixed maximum sequence length. This constrains the
number of objects the model can represent and may bias generation toward a fixed
scene complexity.

Possible improvements:

- Use more flexible variable-length graph generation.
- Condition object count on room size and user requirements.
- Improve stop-token or object-existence modeling.

### 10. Limited Relation Vocabulary

The relation vocabulary is mainly geometric:

- `left of`
- `right of`
- `in front of`
- `behind`
- `above`
- `below`
- close variants

Interior design requires richer relation types:

- `against wall`
- `near window`
- `facing TV`
- `aligned with`
- `reachable from`
- `not blocking`
- `paired with`
- `same style as`

Possible improvements:

- Add architectural, functional, visibility, accessibility, and aesthetic
  relations.
- Use typed graph edges rather than a single flat relation vocabulary.

### 11. Evaluation May Not Capture Design Quality

Metrics based on relation recall or rendered-image similarity may not fully
measure whether a generated room is useful, comfortable, or physically plausible.
A room can satisfy pairwise relations while still failing human design criteria.

Possible improvements:

- Add human preference studies.
- Add path accessibility metrics.
- Add door and window obstruction checks.
- Add collision and clearance metrics.
- Evaluate functional task success, such as reaching the bed, using the desk, or
  opening the wardrobe.

### 12. Poor Extensibility to New Assets and New Categories

InstructScene trains the fVQ-VAE object-feature codebook on furniture objects
collected from the supported room types: bedroom, dining room, and living room.
This makes the object-feature vocabulary specialized for the downstream
InstructScene tasks, but it also reduces extensibility.

If new assets are added but they belong to existing categories, such as a new
`nightstand`, `chair`, or `double_bed`, full retraining may not be required. A
practical path is:

```text
1. Compute OpenShape features for the new asset.
2. Add the asset to the retrieval database.
3. Map it to an existing category.
4. Use the trained fVQ-VAE encoder to quantize its feature if VQ indices are
   needed.
5. Retrieve from the expanded asset pool during inference.
```

This works only if the new asset distribution is close to the existing training
distribution and the object can be mapped to an existing category.

If new categories are added, such as:

- `poster`
- `gaming chair`
- `badminton racket`
- `plant`
- `wall shelf`
- `mirror`
- `sports equipment`

then the original InstructScene pipeline is much less flexible. The object
category vocabulary, semantic graph node classifier, relation vocabulary, layout
decoder, and retrieval filters are all tied to fixed categories. Supporting new
functional categories usually requires dataset preprocessing, vocabulary
expansion, and retraining or fine-tuning of the core models.

This is a real limitation:

```text
InstructScene has poor extensibility for new object types and new asset domains.
```

Possible improvements:

- Same-category asset expansion:
  - Add new assets under existing categories without retraining the whole model.
  - Compute their OpenShape features and include them in retrieval.

- Open-vocabulary retrieval:
  - Replace fixed category retrieval with text/object-feature similarity
    retrieval.
  - Retrieve assets using descriptions such as `Star Wars poster`,
    `ergonomic gaming chair`, or `badminton racket stand`.

- Text-embedding object nodes:
  - Replace closed object category IDs with CLIP/OpenShape text embeddings.
  - Represent object names and styles as language embeddings rather than fixed
    class indices.

- Separate layout category from asset identity:
  - Use coarse categories for layout generation, such as `bed`, `seating`,
    `table`, `storage`, `lighting`, and `decoration`.
  - Use open-vocabulary text descriptions for final asset retrieval or
    generation.

- Hierarchical categories:
  - Represent each object with both a coarse functional category and a fine
    open-vocabulary label.
  - Example:

```text
coarse_type: "seating"
fine_label: "black ergonomic gaming chair"
placement_role: "desk chair"
```

- Lightweight adaptation:
  - Add new categories using LoRA, adapters, classifier expansion, or partial
    fine-tuning instead of full retraining.
  - Freeze most of the learned layout prior and update only new category
    embeddings, output heads, or adapter modules.

The strongest research direction is a hybrid node representation:

```text
node = coarse functional category
     + open-vocabulary text label
     + object feature embedding
     + placement role
```

For example:

```text
{
  coarse_type: "seating",
  text_label: "black ergonomic gaming chair",
  object_feature: CLIP/OpenShape embedding,
  placement_role: "desk chair"
}
```

This would preserve learned layout realism through stable coarse functional
categories while allowing new object names and styles to be introduced through
text/vision-language retrieval.

Potential paper claim:

```text
Unlike InstructScene, which relies on a fixed object vocabulary and requires
retraining when new categories are introduced, our method decouples functional
layout categories from open-vocabulary asset identities. This enables new
objects to be added through text/vision-language retrieval while preserving a
learned layout prior over coarse functional roles.
```

## Bigger Diagnosis

InstructScene is a strong first step for instruction-conditioned 3D indoor scene
synthesis, but its instruction understanding is narrower than the word
`instruction` suggests.

The core limitation is:

```text
InstructScene models spatial relation following,
but not full interior-design reasoning.
```

Real interior design requires a broader pipeline:

```text
human intent
-> functional requirements
-> architectural constraints
-> object selection
-> spatial layout
-> style coherence
-> physical feasibility
```

The current system mainly focuses on:

```text
relation-style text
-> semantic graph tokens
-> object layout and retrieval
```

## Promising Future System Direction

A stronger future system could be modular:

1. LLM planner
   - Converts lifestyle descriptions and design briefs into structured
     requirements.

2. Expanded scene graph
   - Includes furniture, doors, windows, walls, zones, style, material,
     function, and accessibility constraints.

3. Constraint-aware graph generator
   - Generates object categories, object counts, relations, zones, and global
     layout constraints.

4. Layout decoder
   - Produces physically valid positions, sizes, and rotations.

5. Retrieval or generation module
   - Selects or generates 3D assets matching function, style, and object
     features.

6. Critic and refiner
   - Checks collisions, relation satisfaction, door/window obstruction,
     accessibility, style coherence, and user-intent satisfaction.
